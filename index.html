고객 경험 문제를 먼저 정의하고
시스템·정책·UX로 연결해 설계합니다

고객 문의가 반복될 때 저는 처리 속도보다 "시스템이 왜 이 경험을 제대로 지원하지 못하는가"를 먼저 봅니다. 그 답이 API 응답값 구조에 있을 수도 있고, 팝업 UX에 있을 수도 있고, 운영 정책 기준이 없는 데 있을 수도 있습니다.

배송 제약 상품 CS 반복 인입을 분석하다가, 원인이 품절이 아닌 배송일 불일치임을 확인했습니다. 개발이 이미 구분하는 값인데 고객에게 보여주지 않는 것이 문제였습니다. 개발 담당자와 구현 가능 범위를 확인해 "subtext만 다르게 보여주는 정도의 MVP로 가능할까요?"를 제안했고, cartItem API subtext 반영 → 웹 렌더링 최적화 → 3개 상품 확장까지 실행했습니다.

미출고 CS 반복 인입의 원인이 "고객이 미출고 가능성을 구매 전에 전혀 인지하지 못하는 구조"임을 파악했습니다. 경쟁사(오아시스·마켓컬리) UX를 리서치하고 이용약관 20조를 근거로 환불 안내 문구 방향을 정의했습니다.

유기농 식품 이커머스에서 성분에 민감하고 건강·원료에 진심인 고객들의 CS를 운영해왔습니다. 피부 트러블, 알레르기 의심, "믿었던 제품인데"라는 감정이 담긴 문의는 단순 처리 속도만으로는 풀리지 않습니다. 올리브영 뷰티·H&B 고객의 CS 구조도 이와 같다고 생각합니다. 그 경험이 Contact Center Platform 설계에 직접 연결된다고 확신합니다.

서비스 UX 기획 · API 요구사항 정의 상담 데이터 기반 문제 정의 고객-센터-파트너 프로세스 설계 VOC 분석 → 서비스 구조 개선 개발팀 요구사항 협의 · MVP 스펙 정의 계약·약관·정책 기반 운영 기준화
시스템이 아는 걸 고객에게 보여주지 않으면 CS가 된다

개발이 이미 구분하는 API 응답값을 UI에 노출하지 않으면, 고객은 이유를 모른 채 CS에 문의합니다. API 구조를 먼저 파악하고 MVP 스펙을 정의하는 것에서 시작합니다.

실제로 설계하고 개선한 시스템들

시스템 UX 개선 기획 API 구조 기반 요구사항 정의 MVP 제안 · 개발팀 협의

장바구니 배송 제약 상품 UX 개선 · API 응답값 기반 subtext 설계

배송 제약 상품 반복 문의를 분석해 원인이 품절이 아닌 배송일 불일치임을 확인했습니다. cartItem API에서 이미 구분 가능한 값을 활용해 상품별 안내 subtext를 분기 노출하는 MVP 개선안을 개발 담당자와 구현 가능 범위를 확인해 제안하고, 웹 렌더링 최적화 → 3개 상품 확장까지 실행했습니다.

관련 문의 0건 수준 3개 상품 확장 · EC팀 프로세스 개선 제안
상담 시스템 구조 개선 상태값 설계 처리누락 방지

CRM 상태값 라이프사이클 재설계 · 처리누락 방지 구조 개선

1:1 문의 답변 저장 시 상태가 자동으로 처리완료로 전환되어, 추가 확인이 필요한 문의가 완료 건으로 누락될 수 있는 구조를 발견했습니다. 저장(Action)과 상태(State)를 분리하고, 기존 미활용 상태값인 '처리중'을 적용해 중간 처리 상태를 구분했습니다. 이후 고객 화면 노출과 알림 문구까지 함께 정렬해 상담 시스템 상태와 실제 운영 흐름의 불일치를 줄였습니다.

문제 상황 (2026.04.06)
특정 상품에 대해 배송일이 맞지 않아 주문이 안 된다는 CS 문의가 반복 인입되고 있었습니다. 고객들은 이유를 알 수 없어 CS에 문의하는 상황이었습니다.

CS 데이터를 분석하면서 원인이 품절이 아니라 배송일 불일치라는 것을 확인했습니다. 그런데 시스템(cartItem API)은 이미 두 상태를 다르게 구분하고 있었습니다. 고객에게 이 차이를 보여주기만 하면 CS 인입이 사라질 수 있는 상황이었습니다.
문제 정의 · 개발 담당자와 MVP 협의
저는 먼저 API 구조를 파악해 "시스템이 이미 품절과 배송일 불일치를 다른 값으로 구분하고 있다"는 것을 확인했습니다.

이 근거를 갖고 개발 담당자에게 제안했습니다: "시스템에서 이미 구분이 가능하다면, subtext 문구만 다르게 보여주는 정도의 MVP로 가능할까요?"

개발 담당자와 구현 가능 범위를 확인했고, cartItem에 subtext 항목을 추가해 배송 불가 상품에는 배송 제약 안내 문구가 표시되도록 API 응답값 기반 요구사항이 확정됐습니다.
실행 · 반영 · 추가 개선
API 반영 후 웹 화면에서 subtext가 두 줄로 렌더링되는 문제를 발견했습니다. 한 줄로 보여야 깔끔하다고 판단해 문구를 1문장으로 줄이는 UX 최적화를 제안하고 반영했습니다.

반영 완료 후 동일한 배송 제약이 있는 추가 상품(WG029828, 018748)에도 확장 적용을 요청했고, EC팀에는 신규 배송 제약 상품 등록 시 subtext를 함께 입력하는 프로세스 개선도 제안했습니다.
결과
배송 제약 상품 관련 CS 문의가 0건 수준으로 줄었습니다. cartItem subtext API가 배송 제약 상태를 고객에게 직접 안내하는 구조가 만들어졌고, 3개 상품으로 확장 적용됐습니다.

이 작업의 핵심은 운영 문제를 단순 처리하지 않고, API 구조를 직접 파악해 MVP 스펙을 정의하고 개발팀과 협의한 PM 워크플로우였습니다.

문제 상황
1:1 문의에 답변을 저장(Action)하는 순간 상담 상태(State)가 자동으로 '처리완료'로 전환되는 구조였습니다. 추가 확인이 필요한 문의, 파트너 회신을 기다리는 건, 내부 검토가 진행 중인 건이 모두 저장과 동시에 '완료'가 되어버렸습니다.

운영자 입장에서는 실제로 처리가 끝나지 않은 건이 완료 상태로 쌓이면서, 팔로우업이 필요한 문의가 누락될 수 있는 구조였습니다.
문제 정의 · 미활용 상태값 발굴
CRM 시스템 내부를 확인하면서 이미 '처리중' 상태값이 존재하지만 실제로 사용되지 않고 있다는 것을 발견했습니다.

핵심 구조적 문제는 저장(Action)과 상태(State)가 분리되어 있지 않은 것이었습니다. 저장은 작업 동작이고, 상태는 처리 진행 단계입니다. 두 개념이 묶여 있으면 진행 중인 건도 완료로 취급됩니다.

기존에 미활용 상태였던 '처리중'을 적용해, 저장 후에도 '처리중' 상태를 유지할 수 있도록 개선을 요청했습니다.
고객 화면 노출 문제 발견 · 알림 문구 정렬
상태값 변경이 반영된 이후, 고객이 1:1 문의 내역을 조회할 때 '처리중' 상태 텍스트가 그대로 노출되는 문제를 추가로 확인했습니다. 상담 내부 상태값과 고객 화면 노출 메시지가 별도로 설계되어야 하는 지점이었습니다.

고객 화면에는 내부 처리 상태 대신 고객이 인지할 수 있는 안내 문구가 노출되도록 고객 화면 노출 로직 개선을 요청했습니다. 알림 문자 문구도 상태값 변화에 맞게 정렬해 운영 흐름과 고객 인지가 일치하도록 했습니다.
결과

처리 속도가 아니라, 왜 같은 문의가 반복되는지를 데이터로 확인해야 개선 방향을 정의할 수 있다고 판단했습니다.
분석 · 문제 정의
프로모션 오인 VOC 173건을 전수 분류했습니다. 분류 기준은 인입 원인, 발생 채널, 고객 행동 시점, 정책 인지 여부로 설정했습니다.

분류 결과 82%가 동일한 구조적 원인에서 발생하고 있었습니다. 고객이 프로모션 조건을 충족하는지 주문 전 단계에서 확인할 수 없어, 주문 완료 후 조건 미충족을 확인하고 CS에 문의하는 패턴이었습니다. 또한 발생 채널의 94.8%가 모바일이어서 개선 우선순위를 명확하게 정의할 수 있었습니다.
